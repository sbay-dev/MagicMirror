# SPEC-0001 — AI DIRECTIVES (Magic Mirror)

> Executable, code-linked directives that implement the author specifications in
> [`../AUTHOR-LINGUISTIC-SPECIFICATIONS.md`](../AUTHOR-LINGUISTIC-SPECIFICATIONS.md).
> Paths are relative to the repository folder `Migic-Mirror/`.

Solution layout:
- `MagicMirror/` — **web layer**, `NetWasmMvc.SDK` (ASP.NET MVC in WebAssembly / Cepha).
- `MagicMirror.Native/` — **native layer**, MAUI sibling (`Microsoft.NET.Sdk` + `UseMaui`)
  generated by `cepha native-app`, referencing the runtime DLLs in `MagicMirror.Native/lib/`.

---

## D1 — Connect to the MCP server & follow WasmMvcRuntime invariants  ⇐ SPEC-AUTH-001
- The live MCP server `https://wmr-doc.pages.dev/mcp` (JSON-RPC 2.0) and `/api/sdk-knowledge.json`
  were used to derive the architecture. Guardrails followed: single `NetWasmMvc.SDK` SDK line,
  Controllers + Razor views, CMUI styling, no JS SPA, no app-owned `wwwroot/index.html`, MAUI as a
  **sibling** `Microsoft.NET.Sdk` project.
- Web project: `MagicMirror/MagicMirror.csproj`.
- Native bridge: `MagicMirror.Native/Services/CephaMauiBootstrap.cs` (in-process `MvcEngine` +
  `NativeRenderer`, no WebView).

## D2 — AI via a supported Sarmad/Cloudflare model (web gateway + optional fallback)  ⇐ SPEC-AUTH-001
- The original author spec requested gpt-oss-120b. The deployed canonical gateway now reports that
  `@cf/openai/gpt-oss-120b` was deprecated on 2026-05-30, so the executable directive is to use a
  currently supported Cloudflare Workers AI model.
- Web gateway: `MagicMirror/MagicMirror.csproj` sets
  `<CephaSarmadPlatform>true</CephaSarmadPlatform>` and
  `<CephaSarmadModel>@cf/openai/gpt-oss-20b</CephaSarmadModel>`, so a Cloudflare Pages deploy
  exposes `POST /api/sarmad/ask` backed by a supported Cloudflare Workers AI model.
- Client: `MagicMirror.Native/Mirror/SarmadTranslationService.cs`
  - `AskAsync(...)` posts to `{GatewayBaseUrl}/api/sarmad/ask` first, then optional
    `FallbackSarmadUrl` only when explicitly configured.
  - Request contract `{mode, surface, prompt, context, language, model}`, response field `answer`
    parsed by `ExtractAnswer(...)`.

## D9 — Do not crash on Translate  ⇐ SPEC-AUTH-002
- Root cause: an exception on the GraphicsView **render thread** while drawing blocks.
- Fixes:
  - Global handlers: `MagicMirror.Native/App.xaml.cs` `HookGlobalExceptionLogging()`
    (AppDomain + TaskScheduler + WinUI `UnhandledException` with `e.Handled = true`).
  - Render guard: `MirrorDrawable.Draw` wraps `DrawCore`/`DrawBlock` in try/catch → `MirrorLog`.
  - OCR buffer guard: `WindowsMediaOcrEngine` validates `Pixels.Length >= W*H*4` before
    `SoftwareBitmap.CreateCopyFromBuffer`.
  - Diagnostics: `MagicMirror.Native/Mirror/MirrorLog.cs` (writes `mirror.log`) and a headless
    self-test in `MirrorPage` (env `MM_SELFTEST=1`).

## D10 — Do not scramble translated text order (RTL)  ⇐ SPEC-AUTH-003
- Cause: OCR split a visual line into several fragments; translating + right-aligning each
  fragment separately scrambled RTL reading order.
- Fix: `MirrorEngine.MergeRows(...)` merges same-row fragments (overlapping vertical centres) into
  one reading-order line (ordered left→right, text joined, boxes unioned) **before** translation.
- Translation throughput improved by parallel batches in
  `SarmadTranslationService.TranslateBatchAsync` (`MaxParallel`, `BatchSize`).

## D11 — Control translated-text size & arrangement vs. window size  ⇐ SPEC-AUTH-004
- Model: `MirrorSettings.TextScale` and `MirrorSettings.LayoutMode`
  (`OverlayLayoutMode { MatchOriginal, Readable, Uniform }`) in `MirrorModels.cs`.
- Rendering: `MirrorDrawable` applies `TextScale`, `LayoutMode`, a window-adaptive readable floor
  (`MinReadableDip * windowScale`), and grows the text box (anchored at the RTL reading edge) when
  enlarged.
- Live controls (no re-translate): `MirrorOverlayPage` toolbar `A−` / `A+` (`AdjustScale`) and the
  mode-cycle button (`OnCycleMode`); window resize re-renders via `Canvas.SizeChanged`. The size
  step is deliberately visible (`±0.25`) and `MirrorDrawable` applies `TextScale` to the readability
  floor too, so tiny OCR lines resize instead of being pinned by the minimum readable size.
- Persisted settings UI: `MirrorPage.xaml(.cs)` "Text size" slider + "Text arrangement" picker.

## D12 — Faster, smoother live capture  ⇐ SPEC-AUTH-005
- Cheap preview path: `IScreenCapture.CapturePreviewAsync` →
  `WindowsScreenCapture.CaptureBgraScaled` (GDI `StretchBlt` downscale to ≤900 px) + **BMP** encode
  (no PNG deflate). Live preview uses it via `MirrorEngine.CapturePreviewAsync`.
- Smooth drag/resize: `MirrorOverlayPage` pauses live capture while manipulating (`_dragging`
  flag set in `OnDragToolbar`/`OnResizeGrip`, `EndManipulation` resumes with one frame).
- Full-resolution PNG capture is still used only for the on-demand Translate (OCR fallback).

## D13 — Read text without OCR when the window exposes it  ⇐ SPEC-AUTH-005
- Provider: `MagicMirror.Native/Platforms/Windows/WindowsUiaTextProvider.cs` (Windows UI
  Automation). One cached `FindAll` over the foreground window batches property reads; results are
  filtered to the mirror rectangle, oversized containers are skipped, and text-bearing control
  types are kept. Returns capture-relative `OcrTextRegion`s with `Confidence = 100`.
- Build wiring: `MagicMirror.Native.csproj` adds `FrameworkReference Microsoft.WindowsDesktop.App`
  (for `System.Windows.Automation`) **without** `UseWPF` (UseWPF swaps in the WPF XAML compiler and
  breaks MAUI's WinUI XAML).
- Pipeline: `MirrorEngine.TranslateRegionAsync` tries the provider first when
  `MirrorSettings.UseWindowText` is on, and falls back to OCR when it yields nothing. Toggle in
  `MirrorPage` ("Read text from window").

## D14 — Resilient translation (mesh model deprecated)  ⇐ SPEC-AUTH-006
- Root cause: the canonical Sarmad mesh returns HTTP 502
  `{"error":"inference_failed","message":"5028: This model was deprecated on 2026-05-30 ..."}`
  for every request (it ignores the client model field; its upstream default was deprecated by
  Cloudflare). The previous code then silently left every line untranslated while still reporting
  "Translated N/N".
- Historical fix in `MagicMirror.Native/Mirror/SarmadTranslationService.cs` before the v1.0.5
  fallback-hardening gate:
  - `ExtractAnswer` now throws on an `{"error":...}` body so a failed gateway is detected.
  - The original recovery path used **free no-key MT** (`TranslateGoogleGtxAsync` →
    `TranslateMyMemoryAsync`) to avoid silent untranslated output. This behavior is superseded by
    D43: MT may now run only after explicit per-run user confirmation.
  - The AI gateway (configured Sarmad model via the user's deployed gateway) stays **primary**; fallback
    is no longer an automatic safety net.
- Note: to restore the AI path, deploy the web layer (`cepha publish cf`) with a current
  `CephaSarmadModel`, and set the native app's `GatewayBaseUrl` to it.

## D17 — Domain-aware highest-quality translation policy  ⇐ SPEC-AUTH-008
- AI prompt policy: `MagicMirror.Native/Mirror/SarmadTranslationService.cs`
  `BuildTranslationPrompt(...)` instructs the configured Sarmad gateway to silently classify the source
  domain (scientific/academic, religious/sacred, legal, medical, UI, literary, or general) and then
  choose the correct register.
- Scientific/academic mode: requires higher-education formal terminology, conceptual precision,
  consistent rendering of theory/framework/operator/model terms, and non-colloquial Arabic.
- Religious/sacred mode: requires faithful and reverent translation of meaning, preserving doctrinal
  and legal-faith implications without added interpretation, sectarian wording, or replacement of the
  original scripture/hadith text.
- Terminology control: the prompt explicitly preserves acronyms and identifiers such as `LCNS`,
  `CNS`, `CubiCrypt`, `Cubic Neural Statistics`, `GPT`, and `QKV`; it also states that `CNS` must not
  become "central nervous system" unless the source is anatomical/medical.
- OCR-noise handling: `NormalizeOcrNoise(...)` repairs unambiguous OCR errors before translation
  (e.g. `fonnal→formal`, `The01y→Theory`, `govemed→governed`, `mixmg→mixing`, `IIO→no`), and
  `ApplyTerminologyPostEdits(...)` preserves key scientific terms even when the MT fallback is used.

## D18 — Mixed Arabic/English line direction and font construction  ⇐ SPEC-AUTH-009
- Rendering fix: `MagicMirror.Native/Mirror/MirrorDrawable.cs`
  - `PrepareDisplayText(...)` wraps the whole Arabic output line in an RTL bidi isolate and wraps
    embedded English/acronym runs (`CNS`, `LCNS`, `QKV`, etc.) in LTR isolates
    (U+2067/U+2066 plus U+2069), so mixed scientific
    terms keep their internal order without breaking the Arabic line.
  - Mixed RTL/LTR lines switch to `Segoe UI`, a font that reliably covers both Arabic and Latin
    glyphs; pure Arabic lines can still use the detected Arabic family.
  - `EstimateSingleLineWidth(...)` expands the overlay box according to the translated output text,
    not only the original OCR box, preventing long terms/acronyms from being squeezed into a narrow
    source rectangle.

## D19 — Cultural-exchange reading layout and full-window drag  ⇐ SPEC-AUTH-010
- Cultural-exchange goal: the mirror must adapt not only words, but the reading culture: direction,
  line density, mixed-script handling, and user-controlled spacing.
- Dynamic line spacing:
  - `MirrorSettings.LineSpacingScale` stores a user-controlled line-height ratio (`0.85..1.6`,
    default `1.08`) in `MirrorModels.cs` and is normalized in `MirrorSettingsStore.cs`.
  - `MirrorPage.xaml(.cs)` exposes a "Line spacing" slider so Arabic/RTL output can be compact or
    relaxed according to the destination reading culture and the user's eyesight.
  - `MirrorDrawable.LineSpacingScale` uses that value for wrapped translated lines instead of a
    fixed large multiplier, so zooming text does not explode inter-line spacing.
- Collision-free translated blocks:
  - `MirrorDrawable.DrawTranslations(...)` renders blocks in top-to-bottom order and maintains a
    vertical flow cursor (`_flowBottom`).
  - `DrawBlock(...)` moves a block down when its wrapped output would collide with the previous
    block, preferring clipping at the mirror edge over unreadable overlap.
- Full-window manipulation:
  - `MirrorOverlayPage.xaml` attaches `OnDragSurface` to the whole `GraphicsView`, delegating to the
    same move logic as the toolbar, so the mirror can be dragged from any glass area.

## D20 — Preserve document layout, typography, and historical formatting cues  ⇐ SPEC-AUTH-011
- Cultural-exchange principle: the mirror must preserve typographic identity (layout hierarchy,
  serif/sans, heading/body role, and era cues) while adapting the glyphs to the user's target
  language. This treats the page as a cultural artifact, not only a container of words.
- First implementation layer:
  - `MirrorModels.cs` extends `FontInfo` with `DocumentTextRole` (`Title`, `Heading`, `Body`,
    `Caption`) and `TypographyEra` (`ClassicalPrint`, `FormalAcademic`, `ModernDigital`,
    `Newspaper`, `Unknown`).
  - `FontMatcher.Detect(...)` now infers role and era from source box height + text vocabulary
    (e.g. theory/framework/abstract/model/draft → formal academic; UI chrome → modern digital).
  - `FontMatcher.ResolveOverlayFamily(...)` maps the original typographic identity into a
    target-language equivalent family: Arabic formal-academic body text keeps a serif/naskh family
    (`Traditional Arabic`), title/classical print can use `Arabic Typesetting`, and UI/sans text uses
    `Segoe UI`.
  - `MirrorDrawable` asks that resolver for every translated block, so mixed Arabic/English terms do
    not force the whole line into a generic UI font and the source document's typographic signal is
    retained as much as the platform allows.
- Future expansion target: page-level style profiles should aggregate these per-line cues to infer
  document era, column structure, heading hierarchy, and export a fully translated page/image/PDF
  with the original layout preserved.

## D21 — Manual mixed-direction runs and scrollable overflow  ⇐ SPEC-AUTH-012
- Mixed English in Arabic lines:
  - `MirrorDrawable.DrawOutputLine(...)` no longer relies solely on the platform bidi shaping for
    mixed Arabic/English scientific text.
  - `SplitDirectionRuns(...)` divides an output line into RTL and Latin/LTR runs; the renderer places
    those runs manually from the Arabic reading edge while keeping each English run left-to-right.
  - This prevents acronyms and English terms from tearing the Arabic line apart when the target
    paragraph is RTL.
- Scroll overflow:
  - `MirrorDrawable.ContentScrollY` shifts translated content vertically; `MaxContentScrollY` is
    computed from the latest rendered flow height.
  - `MirrorOverlayPage.xaml` adds small `↑` / `↓` controls in the toolbar; `OnScrollUp` /
    `OnScrollDown` adjust the content offset after zooming or when translated lines extend below the
    glass frame.
  - New translation/live-preview passes reset scroll to zero so stale offsets do not hide content.

## D22 — Transparent live-feel body while dragging  ⇐ SPEC-AUTH-013
- UX principle: moving the mirror should feel like moving a real transparent lens over the desktop,
  not like dragging a stale screenshot.
- Windows HWND clipping: `Platforms/Windows/MirrorWindowInterop.SetManipulationClip(...)` applies a
  temporary window region containing only the toolbar and resize grip during manipulation.
- The mirror body is outside the HWND region while moving/resizing, so the desktop shows through
  without a visible color key, black compositor surface, or whole-window alpha fade.
- `EndManipulation()` refreshes/repaints the body while it is still clipped, waits one frame, then
  restores the full region; this prevents a black compositor flash on drop.
- This preserves capture quality: it does not lower preview resolution or keep a misleading frozen
  image during movement.

## D23 — Document/book typography for translated overlays  ⇐ SPEC-AUTH-014
- Translated overlay text must be typeset as a coherent document, not as independently scaled OCR
  fragments.
- `MirrorDrawable.BuildDocumentLayoutProfile(...)` computes one stable page-level typography profile
  per frame from robust median OCR line heights: body, caption, heading, title, line spacing, and
  paragraph gap.
- `ResolveDocumentFontSize(...)` clamps every block to a narrow role-based band around that profile;
  `MatchOriginal` preserves hierarchy only within that band, so noisy OCR boxes cannot create chaotic
  font sizes or rising baselines.
- `DrawBlock(...)` top-aligns multi-line translations, uses dynamic line height, preserves source
  columns with bounded expansion, and uses matte document backings instead of per-line neon panels.
- `DrawDocumentTextBacking(...)` uses an opaque paper-like mask with dark document ink, so the
  original source glyphs are covered under the translation instead of visually colliding with it.
- `DrawOutputLine(...)` renders embedded Latin/scientific runs slightly smaller inside RTL Arabic
  lines so multilingual terms do not dominate line height or disrupt the book-like rhythm.
- `MirrorEngine.MergeParagraphs(...)` reassembles document-like OCR line rows into paragraph boxes
  before translation while preserving `LineHeightHint`; paragraphs translate and wrap as coherent
  book text, but UI Automation/control text remains line-based.

## D24 — Direct manipulation controls and compact mixed-language inline runs  ⇐ SPEC-AUTH-015
- Mouse wheel:
  - `MirrorOverlayPage.HookMouseWheel()` attaches to the WinUI `GraphicsView` platform element and
    maps wheel up/down to `AdjustContentScroll(...)` while translated content is visible.
  - This complements the toolbar `↑` / `↓` buttons and keeps overflow reading natural.
- Double-click/tap:
  - `MirrorOverlayPage.xaml` adds a two-tap recognizer on the glass surface.
  - `OnDoubleTapSurface(...)` translates the current live view when in live mode, and returns to live
    preview when a translated overlay/frozen view is shown.
- Mixed-language spacing:
  - `MirrorDrawable.EstimateInlineRunWidth(...)` uses a tighter width estimator for inline RTL/LTR
    runs than paragraph wrapping uses.
  - Whitespace-only separators no longer reserve a whole font-size box, preventing large blank gaps
    around embedded English acronyms/terms inside Arabic lines.
    - The LTR run regex intentionally matches one English/scientific token at a time rather than a
      multi-word English phrase; this avoids huge clipped runs such as `CubiCrypt/Cubic Neural Statistics`.

## D25 — RTL Latin-term wrap binding and no oversized paragraph masks  ⇐ SPEC-AUTH-016
- `MirrorDrawable.BuildWrapUnits(...)` treats short Latin/scientific tokens in RTL output as part of
  the preceding Arabic wrap unit, so phrases like `إحداثيات CNS` move together instead of letting
  `CNS` start a new line by itself.
- `MirrorDrawable.DrawBlock(...)` no longer paints a union rectangle across the source OCR box and
  the expanded translation box; that union was too large for merged paragraphs and covered unrelated
  page content.
- Body/caption text now uses `DrawOutputHalo(...)`: the translated glyphs get a light document-color
  halo behind the characters rather than a full-width rectangular backing.
- `DrawSourceTextVeil(...)` skips body/caption paragraph boxes and only applies a very faint veil to
  title/heading source boxes, preventing large shaded panels from hiding the rest of the document.

## D26 — User-configurable translation background and ink colors  ⇐ SPEC-AUTH-017
- Persisted defaults:
  - `MirrorSettings.TranslationBackgroundColor`, `TranslationBackgroundOpacity`, and
    `TranslationTextColor` store the translated text halo/background color, its opacity, and the
    translated glyph color.
  - `MirrorSettingsStore.Normalize(...)` clamps opacity and normalizes HEX colors through

- Default settings UI:
  - `MirrorPage.xaml(.cs)` adds HEX entries for translated background/font colors and a slider for

- In-mirror live controls:
  - `MirrorOverlayPage.xaml` adds a compact style bar with background color, background opacity, and

  - `MirrorOverlayPage.xaml.cs` cycles safe document palettes from the mirror itself, persists the

- Rendering:
  - `MirrorDrawable` uses the configured background color/opacity for the glyph halo and heading veil,

  - `MirrorWindowInterop.SetManipulationClip(...)` keeps the new style bar visible during drag/resize
    transparency clipping.

## D27 — Reader-page layout, target-direction normalization, and contextual dictionary review  ⇐ SPEC-AUTH-018
- `MirrorDrawable` treats maximum translation-background opacity as a full reader-page mode: it fills
  the whole mirror with the selected paper color before drawing translated text, making the overlay
  behave like a translated Word/PDF page rather than sparse glyph halos.
- Document blocks use a shared content column derived from body text. For RTL target languages,
  titles, headings, captions, and body paragraphs all align from the reader's right edge instead of
  inheriting the original English source x-position; this prevents title translations from sitting on
  the left while body text follows RTL.
- Mixed Latin tokens are embedded with explicit LTR controls inside an explicit RTL line embedding,
  and short Latin/scientific terms remain bound to adjacent Arabic wrap units so they do not become a
  false line start.
- `MirrorOverlayPage.xaml` increases the appearance style bar height/padding to avoid clipping button
  bottoms.
- `MirrorDrawable.SelectedBlock` and `MirrorOverlayPage` right-click handling highlight the selected
  source/translation block and show a contextual dictionary panel.
- `ITranslationService.ExplainDictionaryAsync(...)` and `SarmadTranslationService` send a strict
  Sarmad prompt requiring domain classification from full document context, at least five
  lexical alternatives, fit/non-fit notes, and a decisive final recommendation.
- Dictionary requests use compact nearby document context and retry with an ultra-compact prompt after
  `413 Payload Too Large`; if the gateway still fails, the panel shows the exact technical reason
  instead of a generic connection message and does not fabricate lexical alternatives without the AI.

## D28 — Render-accurate dictionary selection and movable dictionary panel  ⇐ SPEC-AUTH-019
- `MirrorDrawable.TextHitRegion` records the actual canvas-space line and word rectangles produced by
  `DrawBlock(...)` after document-column normalization, wrapping, scroll, RTL alignment, and text-size
  resolution. Selection no longer guesses from raw OCR boxes that may differ from the visible overlay.
- `MirrorOverlayPage.FindHitRegionAtCanvasPoint(...)` selects the rendered word/line under the pointer,
  with a small nearest-text tolerance. While the dictionary panel is visible, clicks on the glass update
  dictionary selection instead of starting window drag.
- `MirrorOverlayPage.xaml` gives the dictionary panel its own pan gesture and a larger scroll area; the
  panel can be dragged away from the text being inspected.
- `MirrorOverlayPage.FormatDictionaryResult(...)` normalizes dictionary output into separated readable
  sections/numbered alternatives before showing it in the panel.

## D29 — Direct click-to-select text in the mirror  ⇐ SPEC-AUTH-020
- `MirrorOverlayPage.OnCanvasPointerPressed(...)` now treats a normal left click on rendered text as a
  selection action, not only right-click/context-menu input. Blank glass still remains draggable, and
  toolbar dragging is unchanged.
- `MirrorDrawable.RegisterSourceHitRegion(...)` adds source-text OCR boxes as fallback hit regions, so

- `MirrorDrawable.SelectedTextBounds` provides direct selected-hit feedback. Fallback term matching
  and border-style term highlighting must not draw additional boxes when a direct selected-hit mark
  exists.
- The dictionary panel instruction now tells the user to click any word in the mirror to select it.

## D30 — Reliable dictionary movement, source-position selection feedback, and compact gateway payloads  ⇐ SPEC-AUTH-021
- `MirrorOverlayPage.xaml` gives the dictionary card explicit on-card movement buttons (`↖`, `↗`,
  `↙`, `↘`, reset) in addition to a larger drag handle, so the card remains movable even when pan
  gestures are unreliable on the current platform surface.
- `MirrorDrawable.SelectedTextBounds` stores the exact rendered hit selected by the user.
  `DrawSelectedHitHighlight(...)` marks the selected Arabic term/line with a single translucent fill,
  without adding a second source/document outline for the same dictionary selection.
- `MirrorOverlayPage.OnDictionaryExplain(...)` sends only the selected term as `selectedText`; the

  into the selected-text field.
- `SarmadTranslationService.ExplainDictionaryAsync(...)` retries dictionary requests with progressively
  smaller context, down to a selected-term-only prompt. If the gateway still returns an error (for
  example `HTTP 502 BadGateway`), the panel displays that reason first and does not fabricate lexical
  alternatives without a working Sarmad model.

## D31 — Sarmad root-cause fix and stale gateway retirement  ⇐ SPEC-AUTH-022
- Root-cause probe of `https://wmr-doc.pages.dev/api/sarmad/ask` returned
  `HTTP 502` with `5028: This model was deprecated on 2026-05-30. Please use an alternative model.`
- A second probe sent `model: @cf/openai/gpt-oss-20b`, but the same 120b deprecation error returned;
  therefore that gateway is server-side pinned to the retired model and ignores client model
  overrides.

## D35 — User-approved glossary memory and adoptable dictionary proposals  ⇐ SPEC-AUTH-026
- `GlossaryMemoryStore` persists user-approved dictionary selections as local JSON rules in the
  app-data directory. This is a prompt/rule learning layer, not direct fine-tuning of the remote
  Cloudflare model.
- Each stored rule records target language, source text, selected displayed text, accepted
  replacement, nearby context, use count, and timestamps. The store keeps a bounded, recent/frequent
  rule set so prompt payloads remain compact.
- `MirrorOverlayPage` must render an `اعتمد` action for dictionary alternatives that can be parsed
  as candidate replacements. Clicking it updates the selected `TranslatedBlock.TranslatedText`
  immediately and saves the rule through `GlossaryMemoryStore.RememberSelection(...)`.
- `SarmadTranslationService` must retrieve relevant glossary rules for both translation and
  dictionary prompts, inject them as highest-priority user-approved guidance, and apply safe
  post-edit replacements after AI or MT fallback translation when source/context matches.
- If the selected text cannot be found in the current displayed translation, the app must refuse the
  replacement with a visible status instead of guessing a destructive edit.

## D36 — Single selection marker and progressive translation stream  ⇐ SPEC-AUTH-027
- `MirrorDrawable` must not render both direct selected-hit highlighting and fallback term matching
  for the same selection. Direct `SelectedTextBounds` is authoritative; fallback matching must not
  produce repeated boxes across identical words.
- The overlay must preserve the full-line rendering path for Arabic/RTL text. Do not split rendered
  Arabic lines into spaced token columns just to support hit testing.
- `ITranslationService.TranslateBatchProgressiveAsync(...)` yields small ordered translation slices.
  Implementations should keep the existing full-batch method for callers that need a final result.
- `MirrorEngine.TranslateRegionProgressiveAsync(...)` prepares capture/OCR once, yields placeholder
  blocks immediately, then yields updated `MirrorResult` values as each translation slice completes.
- `MirrorOverlayPage.TranslateCurrentViewAsync(...)` consumes the progressive stream and invalidates
  the canvas after each batch so the user sees translation appear gradually instead of waiting for
  the entire document.

## D37 — Typewriter reveal and one authoritative selection rectangle  ⇐ SPEC-AUTH-028
- `SarmadTranslationService.ProgressiveBatchSize` should keep translation slices small, but visible
  delivery must not depend on slice granularity alone.
- `MirrorOverlayPage.QueueTypewriterResult(...)` buffers every progressive `MirrorResult` as target
  text and exposes cloned `TranslatedBlock` values whose `TranslatedText` is revealed by
  `TypewriterTick()` one Unicode text element at a time via `StringInfo.GetNextTextElement(...)`.
- Typewriter reconciliation must preserve placeholders until a real target arrives, reset only the
  block whose target changed incompatibly, and avoid duplicate-target bugs by scanning block indexes
  directly rather than using value-based lookup.
- `MirrorDrawable.DrawSelectedHitHighlight(...)` is the only active selected-word marker path, and it
  uses a single translucent fill rather than a stroked outline. Fallback term matching and block/source
  outline selection must not draw additional yellow boxes for the same active dictionary selection.
- `MirrorDrawable.RegisterSourceHitRegion(...)` must register the source/OCR block geometry exactly
  from capture-derived coordinates, preserving `IsSource`; it must not subdivide source OCR boxes into
  approximate word regions unless a future OCR engine supplies real per-word geometry.
- `MirrorOverlayPage.ResolveSelectionHighlightBounds(...)` must return the exact OCR source block for
  source selections instead of a click-centered approximation.

## D38 — Translation selection must mark its original/OCR counterpart  ⇐ SPEC-AUTH-029
- `MirrorDrawable.SelectedSourceTextBounds` stores a second, source-side marker only when the active
  selection was made from translated text. It is intentionally separate from `SelectedTextBounds` so
  the UI can show "translation + original" without drawing duplicate markers over the same word.
- `MirrorOverlayPage.SelectDictionaryHit(...)` must set `SelectedSourceTextBounds` using
  `ResolveCounterpartSourceBounds(...)` for translated hits and clear it for source-side hits.
- `ResolveCounterpartSourceBounds(...)` must use the capture-derived OCR source block/line for the
  same `TranslatedBlock`. Paragraph-merged blocks must preserve measured OCR source-line rectangles
  through `OcrTextRegion.SourceLines` / `TranslatedBlock.SourceLines`; if measured line provenance is
  unavailable, source-line highlighting must be documented as an inferred visual aid. It must not claim
  source-word precision, because translation reorders and expands terms and that mapping is not
  scientifically grounded.

## D39 — Default high-quality capture for OCR  ⇐ SPEC-AUTH-030
- `MirrorSettings.OcrCaptureScale` defaults to `2.0` and is normalized to `1.0..3.0`, so translation
  captures give OCR larger glyphs by default without changing the low-cost live preview capture.
- `IScreenCapture.CaptureRegionAsync(...)` accepts the quality scale; on Windows,
  `WindowsScreenCapture.CaptureRegionAsync(...)` uses GDI `StretchBlt` with `HALFTONE` into a larger
  BGRA/PNG capture for OCR/Tesseract hand-off.
- `WindowsMediaOcrEngine` must downscale only its internal OCR bitmap when the enhanced capture
  exceeds `OcrEngine.MaxImageDimension`, then scale returned boxes back to the enhanced capture.
- If UI Automation text is used instead of OCR, `MirrorEngine.ScaleTextRegionsForCapture(...)` scales
  those text rectangles into the same capture coordinate space so source highlighting remains aligned.
- Font detection must use the source-size line height, not the upscaled capture height, so a 2x OCR
  image improves recognition without making translated typography look artificially large.

## D43 — Academic-use disclosure and MT fallback gate  ⇐ Expert Council academic release review
- `MirrorSettings.AllowMachineTranslationFallback` remains a privacy preference for MT availability in
  settings, but it is not sufficient to execute MT. When Sarmad fails, returns unusable output, or
  returns the wrong language, the UI must ask for an explicit per-run decision before any third-party
  MT call.
- `MirrorSettings.ForceMachineTranslationFallback` is the per-run execution gate. MT fallback must run
  only when this flag is true after explicit user confirmation; `SarmadTranslationService` must not
  silently reroute gateway failures, wrong-language responses, or unnumbered output to MT.
- MT fallback must be visibly labelled as non-academic quality (`MT fallback (non-academic)` /
  Arabic equivalent), and documentation must state that it sends text to third-party MT services.
- Publisher/confidential workflows must keep MT fallback disabled unless explicitly accepted by the user.
- A valid AI response that preserves a source term unchanged must remain on the Sarmad AI path; fallback
  should be triggered by gateway failure or an unaligned/unparseable response, not by "line changed"
  percentage.
